## FOUNDER POINT 1 — Manual Intervention in Comparison Step

### What changes:
The comparison sheet is auto-generated (AI ranks quotes by landed cost) BUT a procurement
executive must now manually review it before it goes to the approver.

### New flow for Step 10-15:
1. System auto-generates comparison sheet → status: `pending`
2. Procurement executive opens it, reviews each line item
3. They can:
   - Add manual notes on any line item ("Brand X not preferred on this site")
   - Flag anomalies the AI may have missed
   - Override the AI ranking for a specific item with a documented reason
   - Write an overall reviewer note
   - Set their own recommendation (can agree or disagree with AI recommendation)
4. Exec clicks "Mark as Reviewed" → status: `reviewed`
5. Only then does it become available for Procurement Head / Management to approve

### New DB columns added to cps_comparison_sheets:
- `manual_review_status` — pending | in_review | reviewed | sent_for_approval
- `manual_review_by` — uuid of reviewer
- `manual_review_at` — when marked reviewed
- `manual_notes` — overall reviewer notes
- `line_item_overrides` — JSONB array of per-line overrides with reason
- `reviewer_recommendation` — supplier_id chosen by reviewer
- `reviewer_recommendation_reason` — text

### UI rule:
- Approver (procurement_head / management) CANNOT see comparison sheet
  until manual_review_status = 'reviewed'
- Show clear "Awaiting Manual Review" state to approver if not reviewed yet

## FOUNDER POINT 2 — Vendor Names Visible in Comparison Sheet

### Clarification of blind system:
The blind quotation system (Point 3) applies ONLY to:
- Staff reviewing incoming raw quotes (procurement executive during collection phase)
- Quote list page shows only blind_quote_ref, NOT supplier name

The comparison sheet (reviewed by procurement executive and approved by head/management)
SHOWS full vendor names. This is intentional — full transparency at decision stage.

### Rule:
- Quote collection phase → show blind_quote_ref only (e.g. QT-2026-0047)
- Comparison sheet → show supplier name clearly in column headers
- After PO is placed → supplier identity fully revealed everywhere

### Why:
During collection, hiding vendor identity prevents a junior exec from showing
preference. But at comparison stage, the procurement head MUST know which
vendor they are selecting — hiding it there would be impractical and wrong.

## FOUNDER POINT 3 — Blind Quotation System During Collection

### How it works:
1. Supplier submits quote (portal/email/WhatsApp)
2. System assigns a Blind Quote Reference: `QT-2026-0047`
3. This is stored in `cps_quotes.blind_quote_ref` (auto-generated by DB trigger)
4. The supplier_id IS stored in the DB (for PO generation later) but is NOT
   shown to any staff who is reviewing quotes during the collection phase

### Who sees what:
| Role | During Collection | After PO Placed |
|------|-----------------|-----------------|
| Requestor | Nothing | Nothing |
| Procurement Executive (reviewing quotes) | QT-2026-0047 only | Full supplier name |
| Procurement Executive (creating comparison) | Full supplier name | Full supplier name |
| Procurement Head | Full supplier name | Full supplier name |
| Auditor | Full supplier name | Full supplier name |

### New DB columns added to cps_quotes:
- `blind_quote_ref` — auto-generated, unique, format QT-YYYY-NNNN
- `supplier_revealed_at` — timestamp when supplier identity was revealed
- `supplier_revealed_by` — who revealed it

### Supplier gets confirmation:
When supplier submits, they receive confirmation:
"Your quote has been received. Your reference number is QT-2026-0047.
Use this to follow up."

### UI rules:
- Quotes list page: show blind_quote_ref in the main visible column
- Do NOT show supplier name, company logo, or any identifying info in quote review UI
- Comparison sheet: show supplier names (this is post-collection phase)
- After PO is placed: reveal supplier in all views

## FOUNDER POINT 4 — Vendor Self-Registration

### Purpose:
New vendors can register themselves without calling anyone. Keeps the process
fair and open — anyone can apply to supply to Hagerstone.

### New table: cps_vendor_registrations
Status flow: pending → approved (creates cps_supplier record) / rejected

### Registration form fields (PUBLIC — no login required):
**Company Details:**
- Company Name (required)
- GSTIN (optional but recommended)
- PAN (optional)
- Years in Business (number)

**Contact Details:**
- Contact Person Name (required)
- Email (required)
- Phone (required)
- WhatsApp (optional)

**Address:**
- Address (text)
- City, State, Pincode

**Business Profile:**
- Categories they supply (checkboxes: Cement, Steel, Electrical, Plumbing,
  Tiles & Flooring, Paints, Hardware, Safety Equipment, MEP, Other)
- Regions they operate in (checkboxes: Delhi NCR, Mumbai, Bangalore,
  Hyderabad, Chennai, Pan India, Other)
- Business Description (textarea, max 300 chars)
- Reference Clients (optional, textarea)

### After submission:
- System saves with status = 'pending'
- Vendor sees: "Thank you! We'll review your application within 3 working days.
  Your reference: VR-2026-0001"
- status_token is generated (for vendor to check status via a public URL)

### Review by Procurement Head (in /suppliers page, new "Pending Registrations" tab):
- See list of pending registrations
- Click to view full details
- Two actions: Approve or Reject (rejection requires reason)
- On Approve: automatically creates a new cps_supplier record and links it

### Public URL for vendor status check:
- Route: `/vendor/register` — registration form
- Route: `/vendor/status?token=xxx` — status check page (no login needed)

## MOBILE COMPATIBILITY REQUIREMENTS

### Pages that MUST be mobile-first:
1. **Purchase Requisition form** — site engineers fill this on phone at site
   - Large tap targets, single column layout on mobile
   - Easy item search/select
   - Quantity input optimised for touch

2. **GRN Confirmation** — site receiver confirms delivery on phone
   - Checklist-style UI
   - Photo upload for damaged goods
   - Large "Confirm Receipt" button

3. **Delivery Tracker** — site supervisor checks order status on phone
   - Card-based layout
   - Clear ETA display
   - Simple status timeline

### Pages that can be desktop-only:
- Comparison Sheet (complex table, needs large screen)
- Audit Log (admin use)
- Item Master / Supplier Master (admin use)
- RFQ creation (complex form)

### Technical approach:
- Use Tailwind responsive prefixes: `sm:`, `md:`, `lg:`
- Default layout = mobile (single column, full width)
- Desktop = side-by-side, table views
- All buttons minimum 44px tall on mobile (touch targets)
- Bottom-sticky action buttons on mobile forms

## DELIVERY TRACKING — Site Supervisor View

### After PO is placed:
1. Site supervisor is assigned to PO (`site_supervisor_id` in cps_purchase_orders)
2. They receive notification (WhatsApp / in-app)
3. They can see on their mobile:
   - Order details (items, quantities, supplier — revealed after PO)
   - Current status: PO Sent → Acknowledged → Dispatched → In Transit → Delivered
   - LR Number and Transporter name (once dispatched)
   - Expected delivery date
   - Any delays or updates

### Delivery status timeline in UI:
```
✅ PO Generated       — [date]
✅ Supplier Acknowledged — [date]
✅ Material Dispatched — [date] · LR: [number] · [Transporter]
🔄 In Transit         — ETA: [date]
⏳ Delivered          — Pending GRN
```
